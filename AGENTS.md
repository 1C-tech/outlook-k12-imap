# AGENTS.md

本文件定义本项目中 AI Agent 的通用协作规范、开发流程、代码质量要求、测试要求、文档要求与交付标准。所有 Agent 在本项目中工作时，必须优先遵守本文档。

## 核心原则

### 1. 先理解，再行动

在修改代码、创建文件或执行开发任务前，必须先理解当前项目：

- 阅读项目目录结构
- 阅读 README、配置文件和已有文档
- 分析技术栈、框架、代码风格
- 查找相关模块、测试和历史实现
- 尽量复用已有模式，不随意重写

不得在不了解项目上下文的情况下直接开发。

### 2. 尊重用户意图

根据用户当前意图选择行动方式：

- 如果用户要求设计、规划、评审、解释、优化提示词，只输出方案或分析，不修改代码。
- 如果用户明确要求开发、实现、修复、重构、继续，则可以进入代码修改流程。
- 如果需求存在重大歧义或高风险，应先提出待确认问题。
- 如果可以基于项目上下文做出合理判断，应主动推进，不反复打断用户。

### 3. 小步快跑，闭环交付

所有开发任务都应拆成可验证的小任务，每个小任务形成闭环：

1. 明确目标
2. 明确验收标准
3. 阅读相关代码
4. 设计实现方案
5. 编写或更新测试
6. 实现功能
7. 运行测试
8. 修复问题
9. 更新文档
10. 总结交付

## Agent 工作模式

### 产品/需求 Agent

适用于需求分析、产品设计、功能拆解、接口设计、原型设计。

职责：

- 澄清目标用户和使用场景
- 拆解功能模块
- 定义核心流程
- 输出页面或交互原型说明
- 设计数据模型
- 设计 API 契约
- 定义验收标准
- 识别风险和待确认问题

输出内容通常包括：

- 需求理解
- 用户流程
- 功能列表
- 页面结构
- 数据模型
- API 草案
- 验收标准
- 任务拆分
- 待确认问题

### 架构 Agent

适用于技术选型、系统设计、模块拆分、数据流设计、扩展性设计。

职责：

- 分析现有架构
- 识别模块边界
- 设计目录结构
- 定义核心接口
- 设计数据流和状态流
- 评估性能、安全、可维护性
- 提出渐进式实施方案

原则：

- 优先使用项目已有技术栈
- 优先复用已有基础设施
- 避免无必要的大型重构
- 避免引入不必要的新依赖

### 前端 Agent

适用于 Web、移动端、桌面端等用户界面开发。

职责：

- 实现页面结构和交互逻辑
- 保持 UI 与项目现有设计一致
- 处理加载、空状态、错误状态、成功状态
- 保证响应式和可访问性基础体验
- 避免直接渲染不可信内容
- 保持组件清晰、状态可控、样式可维护

前端要求：

- 优先复用已有组件和样式系统
- 不随意引入新的 UI 框架
- 避免过度设计和无关动效
- 保证文字不溢出、不遮挡、不重叠
- 关键交互必须有明确反馈
- 表单必须有校验和错误提示

### 后端 Agent

适用于 API、服务逻辑、数据库、任务队列、权限、安全、集成开发。

职责：

- 实现清晰的服务边界
- 设计稳定 API
- 处理参数校验
- 处理错误与异常
- 编写核心业务测试
- 保护敏感数据
- 保持代码低耦合、高内聚

后端要求：

- 不硬编码密钥、Token、密码
- 不在日志中输出敏感信息
- 配置通过环境变量或配置文件管理
- API 响应结构保持一致
- 错误码和错误信息清晰
- 数据库变更必须可追踪、可回滚或有迁移说明

### 测试 Agent

适用于测试设计、自动化测试、回归测试、质量验证。

职责：

- 根据需求设计测试用例
- 覆盖正常路径、异常路径和边界条件
- 运行项目已有测试
- 为新增功能补充测试
- 记录失败原因
- 验证修复结果

测试优先级：

1. 核心业务逻辑
2. 数据输入校验
3. API 契约
4. 状态流转
5. 权限与安全边界
6. 用户关键路径
7. 回归风险点

## 开发 Loop

每个开发任务必须遵循以下 Loop：

```txt
理解需求
  ↓
阅读上下文
  ↓
制定计划
  ↓
设计测试
  ↓
实现功能
  ↓
运行测试
  ↓
发现问题
  ↓
记录问题
  ↓
修复问题
  ↓
回归测试
  ↓
更新文档
  ↓
交付总结
```

## Git 规范

- 使用 Git 管理项目。
- 可以创建独立项目文件夹。
- 目标仓库路径参考：`https://github.com/1C-tech/outlook-k12-imap.git`
- Commit message 使用以下格式：
  - `feat: 中文描述`
  - `fix: 中文描述`
  - `docs: 中文描述`
  - `test: 中文描述`
  - `refactor: 中文描述`
- 每完成一个可验证任务后提交一次。

## 交付标准

交付前必须确认：

- 相关代码已经实现并保存
- 必要测试已经运行
- 已说明测试结果
- 已说明无法测试的部分和原因
- 已更新必要文档
- 已避免提交日志、数据库、缓存、密钥等本地运行产物

最终回复应简洁说明：

- 修改了什么
- 验证了什么
- 是否已提交
- 是否还有风险或后续注意事项
